// ── LLM · Tool Adapter ─────────────────────────────────────────────────────
//
// HOW NATIVE TOOL CALLING WORKS:
// When an AI model supports "native tools" (also called "function calling"),
// instead of writing [TOOL: ws_read]{...} in plain text, the API returns
// structured data saying "I want to call ws_read with these parameters".
//
// The problem: each provider formats this differently!
//   • Anthropic uses "tool_use" content blocks
//   • OpenRouter uses OpenAI-style "tool_calls" on the message
//   • Ollama doesn't support native tools at all (uses text parsing)
//
// This adapter bridges the gap:
//   1. TOOL_DEFS — master list of all MA tools with their parameter schemas
//   2. buildAnthropicTools() — converts TOOL_DEFS to Anthropic's format
//   3. buildOpenRouterTools() — converts TOOL_DEFS to OpenAI's format
//   4. parseResponse() — extracts tool calls from provider-specific responses
//   5. formatResults() — wraps tool results back for the next API call
//
// WHY TEXT PARSING STILL EXISTS:
// Ollama and some OpenRouter models don't support native tools.
// For those, the AI writes [TOOL: name]{params} in plain text,
// and tools-parser.js handles the extraction.  Both paths eventually
// end up calling the same tool implementations in tools/*.js.
//
// WHAT USES THIS:
//   MA-core.js (via core-chat.js) — builds tool schemas and processes responses
//
// EXPORTS:
//   TOOL_DEFS, buildToolSchemas, buildAnthropicTools,
//   buildOpenRouterTools, parseAnthropicResponse,
//   parseOpenRouterResponse, formatAnthropicToolResults,
//   formatOpenRouterToolResults
// ─────────────────────────────────────────────────────────────────────────────
