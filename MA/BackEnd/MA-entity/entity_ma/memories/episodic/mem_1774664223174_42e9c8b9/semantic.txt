## 1. File Header Comments

Every `.js` file MUST start with a comment header block. This is not
optional. The header is the first thing anyone reads and should answer:

- **What is this file?** (one line)
- **How does it work?** (plain English explanation)
- **What uses this?** (which other files import it)
- **What does it export?** (the public API)

### Template

```javascript
// ── {Group} · {Name} ────────────────────────────────────────────────────────
//
// HOW {CONCEPT} WORKS:
// {Explain the concept in plain English. Use analogies. Imagine you're
// explaining to someone who's never programmed before. 2-6 lines.}
//
// {If there are sub-concepts, use bullet lists:}
//   • {Sub-concept 1} — {brief explanation}
//   • {Sub-concept 2} — {brief explanation}
//
// {If there's a process/flow, use numbered steps:}
//   1. {Step one}
//   2. {Step two}
//   3. {Step three}
//
// WHAT USES THIS:
//   {file1.js} — {why it uses this}
//   {file2.js} — {why it uses this}
//
// EXPORTS:
//   {functionName(params)}  → {return type}
//   {CONSTANT}              — {what it is}
// ─────────────────────────────────────────────────────────────────────────────
'use strict';
```

### The Group Label

The group label tells you which folder/domain the file belongs to:

| Group | Folder | What it contains |
|-------|--------|------------------|
| `NLP ·` | `nlp/` | Text processing algorithms |
| `LLM ·` | `llm/` | AI model communication |
| `Services ·` | `services/` | Business logic and data management |
| `Infra ·` | `infra/` | Infrastructure and system utilities |
| `Tool` | `tools/` | Workspace tool implementations |
| *(none)* | root | Orchestrators (MA-core, MA-routes, MA-workspace-tools) |

### Real Example — Simple Module

```javascript
// ── NLP · BM25 Relevance Scoring ─────────────────────────────────────────────
//
// HOW SEARCH RANKING WORKS:
// Imagine you're looking for a book in a library.  You know the topics
// you care about ("dragons", "magic"), and each book has its own topics.
// BM25 scores how well a book matches your search — higher score means
// better match.
//
// This is the Okapi BM25 algorithm, a proven formula used by real search
// engines.  It accounts for:
//   - How often your search words appear in the document (term frequency)
//   - How long the document is (longer docs get a slight penalty)
//   - A smoothing factor so rare words matter more
//
// TUNING KNOBS:
//   k1 = 1.5   — controls how much extra occurrences of a word matter
//   b  = 0.75  — controls how much document length affects the score
//   avgDL = 8  — assumed average document length (in topics)
//   IDF = ln(2) — simplified inverse-document-frequency (fast, good enough)
//
// WHAT USES THIS:
//   svc-memory.js — ranks memories by relevance when the AI searches
//
// EXPORTS:
//   bm25Score(queryTopics, docTopics, opts?)               → number
//   bm25ScoreWithImportance(queryTopics, docTopics, i, d)  → number
//   K1, B, AVG_DL — the tuning constants (exported for tests)
// ─────────────────────────────────────────────────────────────────────────────
'use strict';
```

### Real Example — Orchestrator Module

```javascript
// ── MA Workspace Tools ───────────────────────────────────────────────────────
// File I/O + web + command tools for task execution.
//
// HOW IT WORKS:
// When the AI wants to DO something (read a file, write code, search the
// web, run a command), it calls a "tool".  This file is the orchestrator
// that coordinates all those tools.
//
// Instead of having ALL the tool logic crammed into one giant file,
// we split it into focused modules in the ./tools/ folder:
//
//   tools-schemas.js  — Zod validation schemas (what params each tool expects)
//   tools-parser.js   — Extracts [TOOL:...] calls from LLM text output
//   tools-fs.js       — Workspace file operations (list, read, write, etc.)
//   tools-entity.js   — Entity creation and memory injection
//   tools-book.js     — Book chunk reading
//   tools-web.js      — Web search, fetch, command execution
//
// This file's job is simple:
//   1. Load each tool module
//   2. When executeToolCalls() is called, parse the text for tool calls,
//      then dispatch each call to the right module
//   3. Collect results and return them
//
// RESILIENCE:
// Each tool module is loaded with safeRequire() — if a module is missing
// or broken, the server still starts.
//
// EXPORTS (public API — same as the old monolith):
//   extractToolCalls(text)                — Parse text for tool calls
//   executeToolCalls(text, opts)           — Parse + execute all tool calls
//   executeNativeToolCalls(toolCalls, opts) — Execute pre-parsed native calls
//   formatToolResults(results)             — Format results for LLM context
//   stripToolCalls(text)                   — Remove tool blocks from text
//   ToolSchemas                            — The Zod schemas for all tools
// ─────────────────────────────────────────────────────────────────────────────
'use strict';
```

## 2. The Comment Voice

Write comments like you're explaining to a smart 10-year-old.

### DO — Use analogies

```javascript
// Think of it like a phone operator — when someone calls "ws_read",
// the operator connects them to the file-reading department.
```

```javascript
// Think of it like a detective's evidence board:
//   • Each sticky note is a NODE (a step, thought, or artifact)
//   • The strings between them are EDGES (connections/relationships)
//   • Thicker strings = stronger connections (higher weight)
```

### DO — Explain the "why", not just the "what"

```javascript
// BAD:
// Normalize the path
const safe = path.normalize(p);

// GOOD:
// Normalize the path to prevent directory traversal attacks.
// Without this, someone could pass "../../etc/passwd" and escape
// the workspace sandbox.
const safe = path.normalize(p);
```

### DO — Use plain words

```javascript
// BAD:
// Implements the Okapi BM25 probabilistic relevance framework

// GOOD:
// Scores how well a document matches a search query (like Google ranking)
```

### DON'T — State the obvious

```javascript
// BAD:
const fs = require('fs'); // require the fs module
let count = 0;            // set count to zero

// GOOD:
const fs = require('fs');
let count = 0;
```

## 3. File Organization Rules

### The 200-Line Rule

If a file is approaching 200+ lines of actual logic, it's time to
consider splitting it. Files over 400 lines should definitely be split.

### The Single-Responsibility Rule

Each file should do ONE thing well. Ask yourself: "Can I describe what
this file does in one sentence without using the word 'and'?"

- ✅ "Scores documents by relevance using BM25"
- ✅ "Extracts tool calls from LLM text output"
- ❌ "Handles config AND bootstrapping AND chat AND token counting"

### The Folder Rules

```
BackEnd/
  MA-core.js              ← Orchestrators live at root
  MA-routes.js
  MA-workspace-tools.js
  core/                   ← Sub-modules of an orchestrator
    core-config.js
    core-bootstrap.js
    ...
  nlp/                    ← Grouped by domain
    nlp-bm25.js
    nlp-rake.js
    ...
  llm/                    ← Grouped by domain
    llm-api.js
    llm-router.js
    ...
  services/               ← Business logic
    svc-memory.js
    svc-tasks.js
    ...
  infra/                  ← Infrastructure utilities
    infra-health.js
    infra-http-utils.js
    ...
  tools/                  ← Tool implementations
    tools-fs.js
    tools-web.js
    ...
  routes/                 ← HTTP route handlers
    route-chat.js
    route-config.js
    ...
```

### What goes where?

| Type of code | Folder | Prefix |
|-------------|--------|--------|
| Text/NLP algorithms | `nlp/` | `nlp-` |
| AI model communication | `llm/` | `llm-` |
| Business logic, data managers | `services/` | `svc-` |
| System utilities, HTTP helpers | `infra/` | `infra-` |
| AI tool implementations | `tools/` | `tools-` |
| HTTP endpoint handlers | `routes/` | `route-` |
| Parts of MA-core.js | `core/` | `core-` |

## 4. Naming Conventions

### Files

- **Prefix matches the folder:** `nlp/nlp-bm25.js`, `services/svc-memory.js`
- **Lowercase with hyphens:** `svc-project-archive.js` not `svcProjectArchive.js`
- **Descriptive but short:** `llm-api.js` not `llm-unified-provider-api-caller.js`

### Functions

- **camelCase:** `extractToolCalls()`, `bm25Score()`
- **Private functions start with underscore:** `_safe()`, `_dispatch()`
- **Boolean functions use is/has/can:** `isConfigured()`, `hasCapability()`

### Constants

- **UPPER_SNAKE_CASE:** `MAX_TOKENS`, `DEFAULT_TIMEOUT`
- **Or descriptive camelCase for complex constants:** `PROVIDER_CAPS`

### Exports

Always put `module.exports` at the very end of the file. List every
export explicitly — never use `module.exports = { ...obj }` from a
variable:

```javascript
// ── Exports ─────────────────────────────────────────────────────────────────
module.exports = {
  extractToolCalls,
  executeToolCalls,
  formatToolResults,
  stripToolCalls,
  ToolSchemas
};
```

## 5. The Orchestrator Pattern

When a monolith gets too big, split it into:
1. A **thin orchestrator** (stays at the parent level)
2. Multiple **focused modules** (go in a subfolder)

### The orchestrator's job:
- Load sub-modules (with safeRequire for resilience)
- Hold shared state (if any)
- Delegate all real work to sub-modules
- Re-export the same public API (consumers don't need to change)

### Rules:
- The orchestrator should be under 300 lines
- It should have almost NO business logic — just wiring
- Every function should be a one-liner that calls a sub-module
- The public API must stay identical (no breaking changes for consumers)

### Example:

```javascript
// ── Orchestrator ─────────────────────────────────────────────────────────────
'use strict';

function safeRequire(modulePath, label) {
  try { return require(modulePath); }
  catch (err) { console.warn(`⚠ Could not load "${label}": ${err.message}`); return null; }
}

const config    = safeRequire('./core/core-config',    'config');
const bootstrap = safeRequire('./core/core-bootstrap', 'bootstrap');
const chat      = safeRequire('./core/core-chat',      'chat');

// Shared state
const state = { config: null, memory: null, entity: null };

// Delegate everything
function boot()   { return bootstrap ? bootstrap.boot(state) : null; }
function handleChat(msg, opts) { return chat ? chat.handleChat(msg, state, opts) : 'Chat module not loaded'; }

module.exports = { boot, handleChat };
```

## 6. The safeRequire Pattern

```javascript
function safeRequire(modulePath, label) {
  try {
    return require(modulePath);
  } catch (err) {
    console.warn(`[${THIS_FILE}] ⚠ Could not load "${label}": ${err.message}`);
    return null;
  }
}
```

### When to use it:
- ✅ Orchestrator files loading sub-modules
- ✅ Dispatcher files loading route handlers
- ✅ Any file that loads many optional modules

### When NOT to use it:
- ❌ Leaf modules loading built-in Node.js modules (fs, path, etc.)
- ❌ Files loading critical dependencies that MUST exist

### Always check for null:
```javascript
const config = safeRequire('./core/core-config', 'config');

// Every call site must handle the null case
function loadConfig() {
  return config ? config.loadConfig(state) : null;
}
```

## 7. Function Comments

### For complex functions, use this pattern:

```javascript
// ─────────────────────────────────────────────────────────────────────────────
// _dispatch(name, params, wp, opts)
//
// Routes a single tool call to the correct module function.
// This is the central switch that connects tool names to implementations.
//
// Think of it like a phone operator — when someone calls "ws_read",
// the operator connects them to the file-reading department.
//
//   name   — normalised tool name (e.g. "ws_read", "web_search")
//   params — validated parameters for the tool
//   wp     — workspace path (for file tools)
//   opts   — execution options (webFetchEnabled, cmdRunEnabled, etc.)
//   Returns: the tool's result string
// ─────────────────────────────────────────────────────────────────────────────
async function _dispatch(name, params, wp, opts) {
```

### For simple functions, a one-liner is fine:

```javascript
/** BM25 relevance score for topic arrays. */
function bm25Score(queryTopics, docTopics, opts = {}) {
```

### Rule of thumb:
- **Under 10 lines:** one-line JSDoc comment
- **10-30 lines:** 2-3 line comment explaining purpose
- **Over 30 lines:** full bordered comment block with params

## 8. Section Dividers

Use these to visually separate logical sections within a file:

```javascript
// ── Section Name ────────────────────────────────────────────────────────────
```

The line should be exactly 78-80 characters wide. Use a consistent
character (─) for the fill.

### Common sections:

```javascript
// ── Imports ─────────────────────────────────────────────────────────────────
// ── Constants ───────────────────────────────────────────────────────────────
// ── Schema ──────────────────────────────────────────────────────────────────
// ── Core Logic ──────────────────────────────────────────────────────────────
// ── Helpers ─────────────────────────────────────────────────────────────────
// ── Exports ─────────────────────────────────────────────────────────────────
```

## 9. What NOT To Do

### ❌ Don't comment every line
```javascript
// BAD
const fs = require('fs');    // import filesystem module
const path = require('path'); // import path module
const data = {};              // create empty data object
```

### ❌ Don't write novels in comments
```javascript
// BAD — this is a comment, not a blog post
// So basically what happens here is that we need to take the input
// and then we process it through a series of transformations and
// each transformation does a slightly different thing and at the
// end we get the output which is what we return...
```

### ❌ Don't put TODO/FIXME/HACK without a plan
```javascript
// BAD
// TODO: fix this later
// HACK: this is ugly but works

// GOOD
// TODO(v2): Replace regex parser with proper AST when Ollama supports native tools
```

### ❌ Don't leave console.log debug statements
```javascript
// BAD
console.log('DEBUG: got here', data);
console.log('xxx', result);
```

### ❌ Don't let files grow past 400 lines
If you're tempted to add more logic to a 350-line file, stop and split
it first. The time you spend splitting now saves 10x the time cleaning
up later.

### ❌ Don't use generic names
```javascript
// BAD
const data = processStuff(input);
const result = doThings(data);

// GOOD
const memories = searchByTopic(queryTopics);
const rankedResults = scoreAndSort(memories);
```

  // ... implementation
